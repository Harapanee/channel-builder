# /channel-refine — システム還元プロセス

仕様書§12に基づく。**勝手に書き換えない。変更は必ず提案→人間承認→適用+記録の順。**

## 0. スコープ確認

変更が**全チャンネル共通の工場側**(video-create等のスキル手順・エージェント定義・src/pipelineツール・スキーマ・共有コンポーネント)に及ぶ場合は、本スキルでの適用後に **/system-refine** でテンプレートへ同期すること(同期しないと改善が「このチャンネル限り」で消える)。

## 0.5 実測データの取り込み(存在すれば必ず読む)

フィードバックの分類・還元先判断の前に、以下があれば読み、根拠に使う:

- `episodes/*/analytics.json`(実測。維持率カーブの脱落点・平均視聴率・CTR手動転記)
- `episodes/*/publish/thumb-test.json`(サムネABの勝ちパターン)

数字は還元先の振り分けに使う: 冒頭30秒の脱落→フック規則(bible §3)、中盤の段差脱落→章構成(§4)・演出(§8)、CTR低迷→サムネ・タイトル規則(§13)。感想と実測が食い違う場合は実測を優先し、その旨をユーザーに明示する。

## 1. 分類

フィードバックが「このエピソード限りの問題」か「再発し得るシステムの問題」かを判定し、ユーザーに明示する。単発ならこのスキルを使わず直接修正してよい。

## 2. 還元先の特定

再発し得る問題の還元先(複数可):

- `channel/bible.md` の該当セクション(人格→§3 / 構造→§4 / 笑い→§5 / アーク→§6 / 体験→§7 / 画→§8 / 素材→§10 / 音→§11)
- `channel/review-checklist.md`(検出観点の追加)
- コアコンポーネント / motionヘルパー(表現力の恒久追加)
- 素材ワークフロー(必須バリアントの追加等)

## 3. 変更提案の提示(適用前に必ずユーザーへ)

以下をまとめて提示する:

- **変更diff**(具体的な追記・修正文)
- **根拠**(元のフィードバック)
- **期待される観測可能な効果**(次のエピソードのどこがどう変わるはずか)
- **既存ルールとの矛盾チェック結果**(bible全文を読み直して確認)
- **行数予算**: bible.md は800行上限。超える場合は同時に削る箇所を提案

## 4. 承認 → 適用 + 記録

**承認済みシステム(`.channel-system.json` の status が "approved")では、bible.md / voice.json / src/scenes/core/ は保護hook(guard-approved.mjs)がEdit/Writeをブロックする。** このためユーザー承認後は必ず次の順で行う:

1. **マーカーを置く**: プロジェクトルートに `.channel-refine-approved`(空ファイルでよい)を作成する。これはhookに「この編集は人間承認済みのrefineである」と伝え、保護対象の一時編集を許可する(status が "building" のうちは保護が無いためマーカーは不要だが、手順として常に置いて構わない)
2. **適用**: 提案どおりに該当ファイルを編集する
3. **CHANGELOG記録**: `CHANGELOG.md` に下記形式で追記する
4. **マーカー削除**: `.channel-refine-approved` を削除する。**削除を忘れると保護が無効のままになる**(.gitignore済みなのでコミットには載らないが、次回以降の保護が効かなくなる)

```markdown
## YYYY-MM-DD
- 契機: <フィードバック原文の要約>
- 変更: <ファイルとセクション、変更内容の要約>
- 期待効果: <次エピソードで観測されるはずの変化>
```

## 5. 効果確認(推奨)

影響範囲が特定ショットに限られる場合は該当部分を再生成・再レンダリングして効果を見せる。全体に及ぶ場合は「次エピソードの人間レビューで効果検証する」旨をCHANGELOGに書く。効かなかった変更はCHANGELOGを根拠に巻き戻す。
